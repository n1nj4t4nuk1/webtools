Day-to-day, the workflow is: write the new strings manually for a handful
of source locales (typically `es`, `en`, `fr`, `de`, `it`, `pt`, `ja`),
then translate the remaining 18 in parallel — usually by farming them out
to subagents in batches of 3.

The setup lives in `nuxt.config.ts`:

```ts
i18n: {
  defaultLocale: 'es',
  fallbackLocale: 'en',
  strategy: 'prefix_except_default',
  lazy: true,
  compilation: { strictMessage: false, escapeHtml: false },
  detectBrowserLanguage: {
    useCookie: true,
    cookieKey: 'tanukibox_lang',
    redirectOn: 'root',
  },
}
```

- `prefix_except_default` → `/mochi` is Spanish, `/en/mochi` is English.
- `lazy: true` → each locale JSON is its own chunk, fetched on demand.
- `escapeHtml: false` + `strictMessage: false` → required because some
  legal-page messages contain `<p>` and `<strong>` and we render them
  via `v-html`.
- `redirectOn: 'root'` → we only auto-redirect from `/`, never from a
  deep link the user shared.

## The three traps

vue-i18n's message compiler reserves three character patterns. If a raw
message contains them and you don't escape, you get a compile-time error
that takes the whole dev server down. All three have happened on this
project; here's the fix for each.

### Trap 1 — bare `@`

`@:` is vue-i18n's **linked message** syntax. Any unescaped `@` followed
by `:` (or sometimes even alone, depending on context) is parsed as a
linked-message reference and breaks compilation.

**Where it shows up:** placeholders like `@example.com`, contact emails
like `hola@tanukibox.com`.

**Fix in plain text (inputs, placeholders):** use the literal
interpolation syntax `{'@'}`:

```json
"suffixPlaceholder": "{'@'}example.com (optional)"
```

vue-i18n resolves this to a literal `@` at render time. The user sees
`@example.com` in the input.

**Fix in messages rendered with `v-html`:** use the HTML entity `&#64;`.
The browser decodes it back to `@` when rendering:

```json
"contact": "<p>For privacy questions: <a href='mailto:hola&#64;tanukibox.com'>hola&#64;tanukibox.com</a>.</p>"
```

### Trap 2 — empty `{}` or `{` followed by non-placeholder text

`{name}` is the standard placeholder syntax. An unmatched or empty `{`
breaks the compiler.

**Where it shows up:** describing punctuation, e.g. "Exclude ambiguous
characters (brackets `{}[]`, etc.)".

**Fix:** rewrite to avoid literal braces in the message. Describe them
in words:

```json
"excludeAmbiguous": "Exclude ambiguous characters (brackets, quotes and similar)"
```

If you absolutely must show a literal `{`, escape it the same way as `@`:
`"{ '{' }"`.

### Trap 3 — HTML inside a message

By default vue-i18n refuses to render `<` or `>` in messages (XSS guard).
We've already turned that off globally with
`compilation.escapeHtml: false` + `strictMessage: false` because the
legal pages embed `<p>`, `<strong>`, `<a>` etc. and render with `v-html`.

**Rules of engagement:**

- **Never** put untrusted/user-provided content inside a translated
  message that you'll render with `v-html`. All translated content is
  ours and is trusted, so this is fine — but don't break the invariant.
- If a message contains HTML, always render it with `v-html`, never
  with `{{ t('…') }}`. The latter would show the angle brackets as
  text.

## How interpolation works

`{name}` is the placeholder syntax:

```json
"match": "Matches {algo}.",
"errorAt": "Error at line {line}, column {col}"
```

Used as `t('hashy.verify.match', { algo: 'SHA-256' })` →
`Matches SHA-256.`.

Plurals exist (`@.plural`) but aren't used in this project. If you need
them later, see vue-i18n's pluralization docs.

## Adding a new string

1. Pick a key. Group related strings under the tool name:
   `slugify.input`, `slugify.actions.copy`, `slugify.errors.empty`.
2. Add it to **all 25** locale files. Don't skip "for later".
3. Translate honestly. If you genuinely don't know a language well,
   leave a TODO comment in the body of the PR rather than
   machine-translating blindly — the user will fix it on review. (Real
   talk: most of these were generated and then sanity-checked by a
   human; they're not perfect, but they're not random either.)
4. If your text contains `@`, `{`, `}`, `<`, `>` or `"`, re-read the
   traps section before saving.

## Adding a new language

1. Create `i18n/locales/<code>.json` with the full key set (copy `en.json`
   as a starting point and translate).
2. Add an entry in `nuxt.config.ts` under `i18n.locales`:
   ```ts
   { code: 'fr', name: 'Français', file: 'fr.json' }
   ```
3. That's it — the language switcher in `app.vue` reads `locales`
   dynamically.

Don't ship a partial locale. Missing keys fall back to English, which
is uglier than not offering the language at all.

## Default and fallback chain

- Default: `es` (no URL prefix).
- Fallback: `en` (used when a key is missing from the active locale).

So a brand-new key only added to `en.json` will appear in English
everywhere. A key only added to `es.json` will appear in Spanish only
and break (key shown literally) in every other language. **Add every
key to every locale.**

## Date and number formatting

We don't use vue-i18n's `$d` / `$n` helpers because tools don't need
locale-aware dates/numbers yet. Tools that do (e.g. timestamp
converter, if it ever exists) should use `Intl.DateTimeFormat` and
`Intl.NumberFormat` with `locale.value` directly.
